# LLM 计费：你到底在为什么付钱？

## 一句话回答你的问题

**是的，如果不做任何优化，输入 token 随轮数线性增长，总花费 ≈ 轮数 n 的平方。**

但实际情况没这么吓人，因为 API 提供商有**缓存机制**，而且工具会做**上下文压缩**。下面一个个讲。

---

## 先搞懂一个基本事实

LLM 是**无状态**的。它不记得上一轮说过什么。

所以每一轮你发的消息，**都必须包含整个对话历史**。工具帮你做的只是"把历史拼接在一起，一次性发给 API"。

---

## 算笔账（假设不做任何优化）

假设每轮：
- 用户说 100 个 token
- 模型回复 200 个 token

| 轮数 | 输入里包含的内容 | 输入 token 数 | 本轮输入费用（按 $0.14/M tokens） |
|------|----------------|-------------|-------------------------------|
| 1 | U₁ | 100 | $0.000014 |
| 2 | U₁ + R₁ + U₂ | 100+200+100=400 | $0.000056 |
| 3 | U₁ + R₁ + U₂ + R₂ + U₃ | 100+200+100+200+100=700 | $0.000098 |
| 4 | ... | 1000 | $0.000140 |
| 5 | ... | 1300 | $0.000182 |
| n | 前面所有轮次 + 本轮 | ≈ 300n - 200 | ≈ (300n-200) × 单价 |

**趋势**：每轮的输入 token 数 ≈ 300 × (n-1)，呈**线性增长**。

**累计总花费**：把所有轮次的费用加起来——

```
总费用 ≈ 单价 × (100 + 400 + 700 + 1000 + ...)
       = 单价 × 等差数列求和
       = 单价 × (150n² - 50n)
```

当 n 很大时 ≈ **150 × 单价 × n²**

**结论：总花费与轮数的平方成正比。** 你说得没错。

---

## 但是——现实世界有三个"救星"

### 救星 1：KV Cache（上下文缓存）—— 同一个前缀不收全价

**这是最关键的机制。**

当 API 处理你的输入时，它要把每个 token 都过一遍神经网络。这个过程包含两步：
1. **Prefill 阶段**：计算所有输入 token 之间的注意力关系（重活）
2. **Decode 阶段**：逐个生成输出 token

如果第 2 轮输入的**前 300 个 token** 和第 1 轮一模一样（就是历史对话），API 提供商会把第 1 轮算好的中间结果**缓存起来**，第 2 轮直接复用。

这意味着：
- 第 2 轮输入 400 token，实际上只有**最后 100 token（新消息）** 需要重新计算
- 前 300 token 走了**缓存命中（cache hit）** 的价格

**DeepSeek V4 Flash 的定价：**
| 类型 | 价格 |
|------|------|
| 缓存未命中（cache miss） | $0.14 / M tokens |
| 缓存命中（cache hit） | **$0.028 / M tokens**（打 2 折） |

回到上面的例子：

| 轮数 | 输入 token | cache miss | cache hit | 实际费用 |
|------|-----------|-----------|----------|---------|
| 1 | 100 | 100 | 0 | 100 × $0.14/M = $0.000014 |
| 2 | 400 | 100 (新消息) | 300 (历史) | 100×$0.14 + 300×$0.028 = $0.0000224 |
| 3 | 700 | 100 | 600 | 100×$0.14 + 600×$0.028 = $0.0000308 |
| n | 300n-200 | ~100 | ~300n-300 | **≈ 线性增长，不再平方** |

**有缓存的情况下，每轮的新增费用基本是常数**（只有新输入的那部分需要全价计算）。

> ⚠️ **注意**：一个 deepseek API key 的缓存作用是全局共享的，但前提是输入前缀**逐字相同**。稍微改一个字，缓存就失效了。所以工具在处理历史消息时，如果做了截断或改写，缓存就帮不上忙。

### 救星 2：上下文压缩（Context Compression）

OpenCode 等工具内置了上下文压缩机制。当对话太长时，它会：
1. 把前面大部分历史**总结成一段摘要**（取代原始对话）
2. 只保留最近几轮详细的对话
3. 去掉不相关的工具调用结果

效果：
```
原始历史：10000 token → 压缩后：2000 token
```

这和"人做会议纪要"是一个道理——你不需要记住会议里每一个人的每一句话，你只需要记住结论和关键数据。

### 救星 3：输出也是要钱的

前面只算了输入，但输出同样计费：
- DeepSeek V4 Flash 输出价格：**$0.28 / M tokens**（比输入贵一倍）
- 输出 token 数每轮 ≈ 模型回复长度，不累计
- 所以输出费用跟轮数成**线性关系**，不平方

---

## 实际操作中的费用长什么样

以一个典型的 OpenCode 调试会话为例：

| 轮次 | 场景 | 输入 token（含缓存） | 费用 |
|------|------|-------------------|------|
| 1 | "帮我看看这个 bug" | 500（全价） | ~$0.00007 |
| 2-5 | AI 问、用户答、修改代码 | 每轮新增 200-500 | ~$0.00003-0.00007/轮 |
| 6 | AI 发现根因，写修复代码 | 新增 800（输出可能较多） | ~$0.0001 |
| 7-10 | AI 继续检查、验证 | 每轮新增 200-500 | ~$0.00003-0.00007/轮 |

**一个完整的调试会话（10 轮）：总花费大约 $0.0005-0.001**

即使一天重度使用 50 轮对话，花费也就在 **$0.01-0.05 左右**。

---

## 可视化总览

```
花费
  ↑                         ✦ 无缓存（平方增长）
  │                      ✦
  │                   ✦
  │                ✦
  │             ✦
  │          ✦
  │       ✦
  │    ✦                            —— 有缓存（线性增长）
  │ ✦                             ···· 输出费用（线性）
  └──────────────────────────────→ 轮数
```

有缓存的情况下：
- **输入费用**（线性的）：每轮新增 ≈ 常数 × 单价 cache miss + 历史 × 单价 cache hit
- **输出费用**（线性的）：每轮 ≈ 模型回复长度 × 单价
- **总费用 ≈ 线性增长，不平方**

---

## 总结

| 你的直觉 | 对/错 | 说明 |
|---------|------|------|
| 输入 token 会越积越多 | ✅ 对 | 历史消息不断追加 |
| 单看输入部分费用越长 | ✅ 对 | 但因为缓存机制，增长慢很多 |
| 总花费 ≈ n² | ⚠️ 理论上对 | 但有缓存命中打 2 折、有上下文压缩，实际接近线性 |
| 越往后对话越贵 | ✅ 对 | 但每轮的边际成本差不多是常数 |

实际结论：**不用担心 n² 问题。缓存机制和上下文压缩让实际费用接近线性增长。一个重度会话的成本通常在几分钱到几毛钱之间。**

## Coding Agent 的特殊问题：Agent Loop 让交互次数暴增

### 普通对话 vs Coding Agent 的核心区别

| 对比维度 | 普通 LLM 对话 | Coding Agent |
|---------|-------------|-------------|
| 典型交互次数 | 1-5 轮 | **10-100 轮** |
| 每轮输入增量 | 几百 tokens | **几千-几万 tokens**（看文件大小） |
| 费用增长模式 | 线性 | **更陡**（每轮带所有历史） |
| 不可控程度 | 低 | **高**——你不知道修一个 bug 要 5 步还是 50 步 |
| 典型费用 | $0.0001-0.001 | **$0.01-0.1+** |

### Coding Agent 的 Agent Loop 是什么

普通对话：
```
用户 → 发消息 → 模型回复 → 结束
```

Coding agent 修一个 bug 的典型过程：
```
用户 → "帮我看看这个 bug"
  ↓
AI 思考 → 决定要读哪个文件
  ↓ 调用 read 工具
工具返回文件内容（500 行代码）
  ↓
AI 思考 → 决定要执行什么命令来复现 bug
  ↓ 调用 bash 工具
bash 返回错误信息
  ↓
AI 思考 → 决定修改哪一行
  ↓ 调用 edit 工具
  ↓
... 这个过程可能重复 10-50 次
  ↓
最终给出结果
```

**每一步都是一次独立的 LLM 调用。** 一个任务下来，10-50 次调用很正常。

### 工具返回结果经常很大——这是关键问题

普通对话：
```
用户发 100 字 → 模型回 200 字
每轮增量：约 300 tokens
```

Coding agent：
```
用户发 50 字 → AI 调用工具读文件
工具返回 500 行代码 → 可能 5000 tokens
本轮增量：5000 tokens（是普通对话的 10-20 倍）
```

跑一个 `grep -r "关键词" ./src` 可能返回几万 tokens 的搜索结果。这些全部计入下一轮的输入。

### 实际算一笔账

以 DeepSeek V4 Flash 为例，假设一个 20 轮的 debugging 任务：

| 轮次 | 事件 | 新增输入 token | 累计输入 token |
|------|------|--------------|-------------|
| 1 | 用户发 prompt | 500 | 500 |
| 2 | AI 读文件 | 3000 | 3500 |
| 3 | AI 跑 bash | 200 | 3700 |
| 4 | AI 分析报错 | 500 | 4200 |
| 5 | AI 读另一个文件 | 5000 | 9200 |
| 6 | AI 跑测试 | 300 | 9500 |
| 7 | AI 修改代码 | 200 | 9700 |
| ... | (重复上述过程) | ... | ... |
| 20 | 最终回复 | ~500 | ~15000 |

**关键问题**：每轮的"累计输入"越来越大，因为每次都要把之前所有内容重新带一遍传给模型。

### 所以实际情况是：缓存救不了 Coding Agent

之前说的"缓存让费用变线性"在 coding agent 场景下要打折扣：

```
第 1 轮：500 tokens → 全价
第 2 轮：3500 tokens → 前 500 hit + 后 3000 miss（全价）
第 3 轮：3700 tokens → 前 3500 中大部分 hit + 后 200 miss
第 5 轮：9200 tokens → 文件内容太大，每次都新增几千 tokens miss
```

**文件内容这类大块新增 token 每次都是 cache miss。** 虽然历史 token 能命中缓存，但新增的文件内容 token 数太大了，导致每轮的实际花费仍然相当可观。

### OpenCode 做了什么来缓解

| 措施 | 做了什么 | 效果 |
|------|---------|------|
| **选择性保留工具返回** | 不是所有工具结果都保留，只保留关键信息（错误信息、行号） | 减少每轮新增 token |
| **上下文压缩** | 历史太长时自动压缩，保留摘要，丢弃原始长输出 | 防止无限膨胀 |
| **输出截断** | bash 或 grep 输出太长时截断 | 防止一次性注入几万 tokens |

### 实际费用参考

| 任务类型 | 循环次数 | 典型费用（DeepSeek V4 Flash） |
|---------|---------|------------------------|
| 简单代码补全 | 3-5 轮 | $0.001-0.005 |
| 读代码 + 小改 | 5-15 轮 | $0.005-0.03 |
| 复杂 bug 调试 | 15-50 轮 | $0.03-0.15 |
| 大规模重构 | 50-100 轮 | $0.15-0.5+ |

一个月重度使用，费用可能在 **几十到几百块人民币** 的量级。

### 控制费用的建议

| 方法 | 怎么做 | 节省多少 |
|------|--------|---------|
| **选便宜模型** | 用 DeepSeek V4 Flash 而不是 Claude Opus | 费用降 50-100x |
| **描述清楚减少走弯路** | prompt 写清楚，减少 AI 反复试探 | 减少 30-50% 轮次 |
| **方向错了及时止损** | 发现 AI 走错方向立刻 `/undo`，不要继续等 | 省大钱 |
| **设置消费上限** | OpenCode Zen 设每月限额 | 防超支 |
| **监控 token 消耗** | 看 API 后台账单 | 心里有数 |

### 总结： Coding Agent 的费用确实比普通对话贵很多

普通对话和 coding agent 的费用差距是 10-100 倍，而且 coding agent 的费用存在**不可预测性**——你不知道修一个 bug 到底要多少轮。

**核心建议**：重度使用时关注账单，设置消费上限，方向错了立刻止损不要硬撑。

## 附录：Prefill 和 Decode 详细解释

### 为什么 KV Cache 让缓存命中变成可能

Transformer 处理 token 的核心是**注意力机制**——每个 token 要和前面所有 token 计算关系（两两看脸色）。

这产生了 K（Key）和 V（Value）两个向量，每个 token 一组。计算完之后，服务器把每组 K 和 V 存起来，这就是 KV Cache。

### Prefill 阶段：处理所有输入

当你发一段话给 API，比如说 500 个 token：

```
500 个 token 要两两算 Attention
即 500 × 500 = 250,000 次计算

每算完一个 token，它的 K 和 V 就被存进缓存

这 500 个 token 的计算是"一次性"的，这就是 Prefill
```

Prefill 的代价被计入"输入 token 费用"里——你付的钱覆盖了这 250,000 次计算。

### Decode 阶段：逐个生成输出

模型开始吐字了：

```
生成第 1 个输出 token：
- 它的 K 和 V 要新算（一 handful）
- 它要和 Prefill 算好的 500 个 K 做 Attention（直接复用缓存）
- 比 Prefill 便宜很多

生成第 2 个输出 token：
- 现在缓存里多了第 1 个 token 的 K 和 V
- 它要和 Prefill 算好的 500 个 K + 第 1 个 token 的 K 做 Attention
- 以此类推，每生成一个 token，复用之前所有 token 的缓存

这逐个生成的过程就是 Decode
```

### 缓存对账单的影响

KV Cache 节省的是**服务器算力**。对你的账单：

```
你的账单 =
  输入 token 数 × 输入单价（Prefill 覆盖全部输入）
+ 输出 token 数 × 输出单价（Decode 逐个生成）

KV Cache 不改变你看到的价格标签
它让 DeepSeek 的算力成本更低，所以 DeepSeek 可以给出更低的价格
```

DeepSeek 的 cache hit 定价是 miss 价格的 1/5（$0.028 vs $0.14），这就是它把节省的算力让利给用户的方式。
