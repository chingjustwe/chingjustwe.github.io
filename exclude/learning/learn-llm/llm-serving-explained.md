# LLM 底层原理：模型显存、KV Cache 与大规模服务

> ⚠️ **阅读说明**：本文整合了 Transformer 架构中的显存占用、KV Cache 与大模型服务工程实践。目标读者为零基础读者。

---

## 一、模型权重 vs KV Cache：厨师与便签纸

### 类比：厨师的知识 vs 做菜时的临时笔记

```
模型权重（Weights）= 厨师脑子里记住的全部做菜知识
                    学会了就永久不忘，所有菜共用这一套知识

KV Cache            = 厨师做某道菜时在便签纸上写的东西
                    这道菜做完就扔，下道菜重新写
                    每个用户、每个对话都要重新写
```

### 具体数据对比

以 DeepSeek V4 Flash 为例（70B 参数，MoE 架构，实际每次激活 13B）：

| | 模型权重 (Weights) | KV Cache |
|---|---|---|
| **内容** | 神经网络所有参数（几十亿个数字） | 每个 token 的 K 和 V 向量 |
| **大小** | 固定：70B 参数 × 2 bytes ≈ 140GB | 动态：约 32KB / token |
| **1M token 需要多少** | 固定 140GB（不变） | ≈ 32GB |
| **生命周期** | 加载后常驻显存，直到服务重启 | 每个 Session 独立，生成完就释放 |
| **归属** | **所有用户共用这一份** | **每个用户、每个对话独立一份** |
| **类比** | 整本菜谱书 | 做某道菜时的便签纸 |

### 在 GPU 显存里的真实布局

```
┌────────────────────────────────────────────────┐
│              单张 GPU 显存（80GB）                │
├─────────────────┬──────────────────────────────┤
│  模型权重        │        KV Cache             │
│  ~35GB（INT4）  │    按需分配，~4-16GB        │
│  所有用户共享     │    每个 Session 独立        │
├─────────────────┴──────────────────────────────┤
│                    其他开销                       │
│     推理中间结果、临时缓冲等                    │
└─────────────────────────────────────────────┘
```

### 两者的关系

跑一个推理请求，必须同时有：

```
Step 1: 把模型权重加载进显存（一次性成本，无论多少用户）
Step 2: 用户发消息 → 按需分配 KV Cache 显存
Step 3: 生成完毕 → KV Cache 释放（模型权重不动）
Step 4: 新消息进来 → 重新分配 KV Cache
```

## 二、大模型服务的硬件问题：亿级用户怎么办

### 首先要破除的误解

你担心的公式："亿级用户 × 每人占用 100GB = 天文数字"

这个担忧基于一个错误假设：**所有用户同时占用。**

```
实际用户使用模式：

用户在线但没在生成  ┌──── 几乎不占显存 ────┐
                     ↓
用户输入 → AI 生成  │ 占 KV Cache，但只有几秒-几分钟
（整个过程 30-60 秒）  │
                     ↓
用户去看结果          ┌──── KV Cache 释放 ────┘

平均每天用 5 次，每次 30 秒
= 每天 150 秒 = 2.5 分钟的实际生成
= 99.97% 的时间这个用户不占任何显存
```

**核心事实：不是"亿级用户同时占用"，而是"每天亿级访问，并发 Active 用户只有几万到几十万"。**

## 三、大厂七层解法

### 解法 1：模型权重所有用户共享（最根本）

不管多少用户，模型权重在 GPU 里只有一份。

```
不管有多少用户同时在聊天
DeepSeek 的模型权重：只有一份

用户 1、用户 2、...用户 N 都调用同一份模型权重
```

所以"每个用户都要一个完整模型"本身就是个伪命题。

---

### 解法 2：连续批处理（Concurrency Batching）

把多个用户的请求凑成一组，一起算。

```
传统方式（低效）：
  用户1的请求 → 算完 → 用户2的请求 → 算完 → ...
  GPU 大部分时间在等待，利用率 < 30%

连续批处理（高效）：
  用户1-100的请求 → 凑成一批 → 一起算
  GPU 利用率提升到 70-90%
```

10 个用户凑一批，GPU 利用率翻 3 倍，等于硬件成本降了 3 倍。

---

### 解法 3：Paged Attention（vLLM 的核心技术）

**问题**：传统 KV Cache 必须连续存放，像一整张便签纸。分配大了浪费，分配小了放不下。

**解法**：把 KV Cache 分成 4MB 一块的"页"，用操作系统虚拟内存的思路管理。

```
传统方式（连续分配）：
  用户A的缓存：第1块  第2块  第3块  第4块（必须连续，中间空着也不能给B用）
  用户B的缓存：等待A释放

vLLM方式（页式管理）：
  用户A的缓存：第1页  第5页  第8页  （按需分配，不用连续）
  用户B的缓存：第2页  第3页  第9页  （并行使用，不等A）
  显存利用率从 20-30% → 80%+
```

效果相当于：GPU 数量不变，能同时服务的用户数翻 3-4 倍。

---

### 解法 4：INT8 / INT4 量化

把模型权重的精度压缩，减少显存占用：

```
bfloat16（正常精度）：每个数字占 2 bytes
INT8（8位整数）：     每个数字占 1 byte      →  省 50% 显存
INT4（4位整数）：     每个数字占 0.5 byte    →  省 75% 显存
```

精度略微下降，但推理效果差距很小。这是所有大厂的通用做法。

---

### 解法 5：KV Cache 选择性丢弃和压缩

没有人真的给每个用户无限制存 1M token 的 KV Cache。实际做法：

```
方案 A：直接截断
  → 超出上限的部分直接丢弃，只保留最近的内容

方案 B：滑动窗口
  → 只保留最近 N 个 token 的缓存（类似人脑记忆）

方案 C：Summarization 压缩
  → 把长历史总结成一段摘要，用摘要替代原始历史

方案 D：Sparse Attention
  → 只关注重要的 token（如关键词、命名实体）
  → 不重要的 token 不建完整的 K 和 V
```

---

### 解法 6：MoE 架构（DeepSeek V4 Flash 的绝招）

> ⚠️ **MoE 常见误解澄清**：MoE 主要省的是**算力（FLOPs）**，不是**显存（VRAM）**。所有专家仍需完整加载进显存，因为不同 token 可能路由到不同的专家。

#### 为什么所有专家都要在显存里？

MoE 的专家选择不是"选一次用到底"，而是**每个 token 动态路由**：

```
同样的输入："我喜欢吃苹果"

Token "我"   → 路由到专家 [E7, E42]（负责单字和语气）
Token "喜"   → 路由到专家 [E3, E15]（负责情感表达）
Token "欢"   → 路由到专家 [E3, E15]（同上）
Token "吃"   → 路由到专家 [E22, E58]（负责动作/消耗）
Token "苹"   → 路由到专家 [E31, E77]（负责具体名词）
Token "果"   → 路由到专家 [E31, E77]（同上）
```

如果你只加载了 E7 和 E42，当"苹"字过来要路由到 E31 时，E31 不在显存里，你没法跑。

#### 类比：200 个专科医生的急诊科

```
医院总共有 200 个专科医生（284B 参数）
每个病人来了，前台根据症状分配 1-2 个医生（激活 13B）
这个病人看心内科（E7），下个病人看骨科（E31）

问：你只需要医院里有心内科医生就够了？
答：不行。因为你不知道下个来的是骨科还是神经科。

你必须让所有 200 个医生同时在岗（加载所有 284B 参数）
但每个病人只需要 1-2 个医生（只激活 ~13B 参数）
```

#### 数据和公式

```
DeepSeek V4 Flash（MoE）：
  总参数：284B
  激活参数：~13B（/ token）
  精度：INT4

显存需求（模型权重）：
  bf16：284B × 2 bytes ≈ 568GB  ← 总和很大
  INT4：284B × 0.5 bytes ≈ 142GB
  ✓ 需要全部 284B 的显存（总和显存）

算力需求（每 token）：
  Dense 模型（70B）：每 token 要算全部 70B 参数
  MoE 模型（284B/13B）：每 token 只算 ~13B 参数
  ✓ 节省的是算力，不是显存
```

#### 但有个省钱的办法：Expert Parallelism

虽然**总和**需要 284B 的显存，但可以**把不同专家放在不同 GPU 上**：

```
把 256 个专家分散到 32 张 GPU：

GPU 0：专家 [E0-E7]  + Attention 层（每张卡都有）
GPU 1：专家 [E8-E15] + Attention 层
GPU 2：专家 [E16-E23] + Attention 层
...
GPU 31：专家 [E248-E255] + Attention 层

Token "苹" 路由到 E31 → 在 GPU 3 上
→ 把"苹"的中间结果通过网络发到 GPU 3
→ GPU 3 算完，结果发回来
```

每张 GPU 只存 ~9B 专家参数 + Attention 层，一张卡只需要 ~16GB 存权重。这是**真正让 MoE 在工程上省钱的方式**（水平扩展）。

#### MoE 省与不省的总结

```
┌────────────────────────────────────────────────────┐
│ MoE 省了什么，没省什么                             │
├────────────────────┬───────────────────────────────┤
│ ✅ 省算力（FLOPs） │ 每 token 只激活 13B/284B      │
│ ✅ 省单卡显存      │ Expert Parallelism 下每张卡    │
│                    │ 只存部分专家（但需要很多卡）    │
│ ❌ 不省总参数量    │ 284B 就是 284B，总和这么大     │
│ ❌ 不省磁盘空间    │ 模型文件 ~150-570GB            │
│ ❌ 不省总显存      │ 所有 GPU 显存加起来仍要装下    │
│                    │ 全部 284B 参数                 │
└────────────────────────────────────────────────────┘
```

### 解法 7：分层推理 + 廉价模型路由

不是所有请求都用最强的模型：

```
简单请求（"帮我写个 hello world"）
  → 路由到便宜的小模型（Qwen-7B / DeepSeek-V3）
  → 成本低，速度快，效果好

中等请求（"帮我优化这个 SQL"）
  → 路由到中档模型（DeepSeek-V4-Flash）
  → 成本中等

复杂请求（"帮我重构整个后端架构"）
  → 路由到最强模型（Claude Opus / GPT-5）
  → 成本高，但效果最好
```

80% 的简单请求被便宜模型处理，只有 20% 的复杂请求用最贵的模型。

## 四、分布式推理中的 Cache Miss 问题

### 问题场景

```
用户第一次对话（Turn 1）
  → 负载均衡器分配到 GPU-A，KV Cache 存在 GPU-A
  → 对话结束，用户去忙了

过了一小时，用户回来了（Turn 2）
  → 负载均衡器分配到了 GPU-B（GPU-A 现在很忙）
  → GPU-B 上没有 Turn 1 的 KV Cache
  → 要么重算 Turn 1，要么降级

如果运气好分配回 GPU-A：
  → 但 GPU-A 可能因为其他用户把缓存 LRU 淘汰了
  → 还是 miss
```

### 解法 1：Session Affinity（粘性路由）

负载均衡器维护一张映射表：

```
session_id → gpu_id

{"sess_abc123": "gpu-07", "sess_def456": "gpu-15", ...}

Turn 1：用户进来 → 分配到 GPU-07
         → 记录 sess_abc123 → GPU-07
Turn 2：用户回来 → 查找 sess_abc123 → 强制路由回 GPU-07
```

**问题**：如果 GPU-07 负载满了怎么办？

实际做法更灵活：**同一 Pod（GPU 组）内做 Session Affinity**

```
一组 GPU [GPU-07, GPU-08, GPU-09, GPU-10] 组成一个 Pod

Turn 1 → Pod-6（GPU-07 处理）
Turn 2 → 仍在 Pod-6 内，分配到 GPU-09
         → GPU-07 与 GPU-09 之间有 NVLink/InfiniBand
         → KV Cache 可以通过高速互联传输（微秒级延迟）
```

### 解法 2：KV Cache Offloading（卸载到 CPU 内存/磁盘）

等不及重算又不想丢缓存：

```
GPU 显存（HBM）    → 最快但贵，容量小 ← 缓存默认存放处
  ↓ eviction
CPU 内存（DDR）    → 中速，容量大（~512GB/1TB）← 冷数据卸载
  ↓ eviction
NVMe SSD           → 慢，容量巨大 ← 最冷的数据

用户 Turn 1 结束 → KV Cache 从 GPU → CPU 内存
用户 Turn 2 回来 → KV Cache 从 CPU → GPU（几十毫秒 vs 重算几秒）
```

**代价**：CPU→GPU 传输有延迟，但比重新 Prefill 快得多。

### 解法 3：Prefix Cache 跨用户共享

不仅是你的 Session 内的 cache 能复用，**别人的也能**。

```
用户 A 和用户 B 都用同一个系统提示词：

System: "你是一个编程助手，使用中文回复..."

用户 A 问："帮我写个快排"     ↓
用户 B 问："帮我写个冒泡排序"  ↓
                  ↓
系统提示词部分完全一样 → KV Cache 跨用户共享

用户 A 先触发 Prefill：
  [System] → [你] → [是] → [一] → [个] → ... → KV Cache 缓存

用户 B 后来请求：
  检测到系统提示词完全匹配 → 直接复用用户 A 的缓存
  只需要算"帮我写个冒泡排序"那部分
```

这就是 API 定价中 **Cache Hit / Cache Miss** 的本质：

- Cache Hit：系统提示词（加上部分用户前缀）已经有人算过，直接复用
- Cache Miss：这部分内容第一次出现，没有缓存

### 解法 4：Automatic Prefix Caching（vLLM 原生支持）

不是精确匹配整段文字，而是**按 token 序列自动匹配**：

```
输入："请你帮我写一个快速排序算法"

系统检测到：
  "请你帮我写一个" → 缓存命中（其他用户之前问过类似问题）
  "快速排序算法"  → 可能未命中（具体内容有差异）

实际效果：
  前缀匹配越长 → 省越多算力
  常见开头如"你是一个编程助手..." → 命中率极高
```

### 解法 5：Session Timeout + 分层缓存策略

```
用户 A 结束对话：
  Turn 1 → Turn 2 （5分钟内回来）
    → 大概率分配回同一 GPU（缓存仍在 GPU 显存）
    → 成本：极低（几乎 0 额外开销）

  Turn 1 → Turn 2 （30分钟-1小时）
    → 缓存可能被卸载到 CPU 内存
    → 回来时重新加载到 GPU
    → 成本：中（几十毫秒传输延迟）

  Turn 1 → Turn 2 （超过超时时间，一般 5-15 分钟）
    → 缓存被释放
    → 下次完整 Prefill
    → 成本：高（几秒算力）
```

### 各种场景的成本对比

```
场景                                              | 代价
──────────────────────────────────────────────────|──────────
Turn 2 路由到同一 GPU，KV Cache 还在            | 接近 0
Turn 2 回到同一 Pod，缓存通过高速互联传输        | 微秒级延迟
Turn 2 在不同 Pod，缓存从 CPU 加载               | 几十毫秒
Turn 2 在不同 Pod，缓存已被淘汰                  | 完整 Prefill（几秒）
Turn 2 前缀匹配缓存（Cache Hit 定价）            | ~$0.028/M token
Turn 2 完全无缓存（Cache Miss 定价）             | ~$0.14/M token
```

### 核心思路

```
不是试图避免所有 Cache Miss
而是把 Cache Miss 分成"贵和便宜"两种

便宜的 Cache Miss：
  系统提示词已缓存（复用别人的）
  → 只算用户独特的部分
  → 大多数人属于此类

贵的 Cache Miss：
  整个前缀完全没有人算过
  → 相对较少（系统提示词基本不变）

API 定价已经反映了这个现实：
  Cache Hit = 大多数情况
  Cache Miss = 少数情况
  平均成本已摊入定价
```

## 五、亿级用户的真实资源估算

```
假设（2026年典型数值）：
  - 注册用户：1 亿
  - 每天活跃用户（DAU）：1000 万（10%）
  - 同时在生成的用户：~10 万（DAU 的 1%）

每用户峰值显存占用：
  - 模型权重（所有用户共享）：35GB（INT4 量化后，仅一份）
  - KV Cache（生成时）：~8GB（按需分配）

10 万并发用户需要的 KV Cache：
  10万 × 8GB = 800TB 显存

加上七层武器的优化：
  ✅ Paged Attention         → 显存利用率 3x
  ✅ 连续批处理            → 并发效率 10x
  ✅ 非活跃用户不占缓存     → 并发降到 1/10
  ✅ 滑动窗口/Summarization → 单用户实际占用降 5x

最终需要的 GPU 数量：
  ~500-2000 张 H100/A100（总成本约 10-40 亿人民币）
```

对比单用户自己跑一个 70B 模型（需要 2-4 张 H100 = 20-40 万人民币），云服务通过共享权重 + 批处理 + 调度，把成本均摊到了每个用户头上。

**这就是大模型服务的本质：用软件工程和调度算法，让一个硬件集群服务海量用户，而不是给每个人配一套硬件。**

## 六、总结

| 问题 | 答案 |
|------|------|
| 模型权重和 KV Cache 区别 | 权重是"知识"（常驻、共享），KV Cache 是"便签纸"（临时、独立） |
| MoE 省显存还是省算力？ | **省算力不省显存**。所有 284B 参数必须在显存里，但每 token 只算 13B |
| Expert Parallelism 的作用 | 把不同专家放不同 GPU 上，每张卡显存需求降到 1/N，靠 InfiniBand 互联 |
| 跨 GPU Cache Miss 怎么解决 | Session Affinity（粘性路由）+ KV Offloading + Prefix Cache 跨用户共享 |
| 为什么 Prefix Cache 能跨用户 | 系统提示词一样（如"你是一个编程助手"），不同用户可复用 |
| 亿级用户需要天文数字硬件？ | 不是同时占用。通过批处理、共享权重、缓存复用，实际是几千张 GPU 的量级 |
| DeepSeek 怎么做到的 | 七层组合拳：共享权重 + 连续批处理 + Paged Attention + 量化 + KV 压缩 + MoE + 分层路由 |
| 最关键的技术突破 | Paged Attention（vLLM）让显存利用率从 20% → 80%，同样硬件 serve 4x 用户 |

## 附录：算力与时间量级速查

### FLOPs 公式

```
Prefill FLOPs  ≈  6 × P_activated × n  +  4 × d_hidden × n²

  Linear 项（6Pn）：所有激活参数 × token 数
    → 复杂度 O(n)，短文本占主导

  Quadratic 项（4dn²）：注意力分数计算
    → 复杂度 O(n²)，长文本占主导
```

```
Decode FLOPs（每步）  ≈  6 × P_activated  +  4 × d_hidden × n

  每步只有 O(n) 的注意力 + 固定激活参数
  → 比 Prefill 每 token 的计算量小得多
  → 主要受显存带宽限制，不是算力限制
```

### 瓶颈切换的直觉

```
Linear 主导 ←──→   Quadratic 主导
        ↓                  ↓
    短文本             长文本（>50K token）
    算力 ∝ n          算力 ∝ n²

  n 的增长影响：
    n × 2 → Linear 项 × 2，Quadratic 项 × 4
    n × 4 → Linear 项 × 4，Quadratic 项 × 16
    n × 10 → Linear 项 × 10，Quadratic 项 × 100
```

### Prefill 时间速查（13B 激活，FP8，H100）

使用多卡时时间近似按比例缩减（沟通开销使实际略高）。

| 输入长度 | 1×H100 | 8×H100 | 生产常见场景 |
|---------|--------|--------|------------|
| 1K | ~0.05-0.2s | — | 简单问答 |
| 4K | ~0.3-0.8s | ~0.05-0.1s | 短对话 |
| 16K | ~1-3s | ~0.2-0.5s | 中等文档 |
| 32K | ~3-8s | ~0.5-1.5s | 代码仓库分析 |
| 64K | ~8-20s | ~1-3s | 技术文档分析 |
| 128K | ~20-60s | ~3-10s | 论文审阅 |
| 1M | ~10-30 min | ~1-4 min | 整库分析 |

> 注：实际延迟因模型架构、量化精度、FlashAttention 实现、显存带宽利用率而异。上述是经验范围，适合做心理模型。

### Prefill vs Decode 对比

| | Prefill | Decode（每步） |
|---|---|---|
| **处理什么** | 一次性处理所有输入 token | 每次生成 1 个输出 token |
| **复杂度** | O(n × P + n² × d) | O(P + n × d) |
| **瓶颈** | 计算密集（GPU 算力） | 带宽密集（显存带宽） |
| **FLOPs 利用率** | 30-50% | 2-5% |
| **4K 耗时** | ~0.5s | 每步 ~8ms，全流 ~0.4s |
| **128K 耗时** | ~40s | 每步 ~36ms，全流 ~5s |
| **谁更耗总时间** | n>1K 时 Prefill 占主导 | 输出长时 Decode 占主导 |

> 对 4K 短文本，Prefill（0.5s）反而比完整 Decode 4K（0.4s）慢，因为 Decode 每步只算一个 token 带宽需求小。对 128K 长文本，Prefill（~40s）远超 Decode（~5s），因为 Quadratic 项爆炸。**这解释了为什么大模型厂商更在意优化长文本 Prefill 而非 Decode。**
