# Transformer 是什么？—— 大白话入门

> ⚠️ **阅读说明**：本文目标是让零 AI 背景的人理解 Transformer 的核心原理。为了好理解，有些地方做了简化，不完全严谨。

---

## 先说清楚：Transformer 是深度学习时代最重要的发明

2017 年，Google 发了篇论文叫《Attention Is All You Need》，提出了 Transformer。这个架构一出来就横扫了整个 AI 领域——翻译、写作、代码、对话、图片生成……背后都是 Transformer。

它是 GPT、Claude、DeepSeek 等所有大模型的**底层引擎**。理解它，才能理解 AI 为什么能"生成"文字。

---

## 一个核心类比：做阅读理解题

想象你读了一篇很长的文章，现在老师问你：

> **"根据这篇文章，接下来最可能发生什么？"**

你的思考过程是这样的：

```
第 1 步：回顾文章内容
  → 文章讲的是：主角在森林里迷路了，天气越来越冷

第 2 步：结合上下文猜测
  → 天气冷 + 迷路 = 最可能生火取暖，或者找到出路

第 3 步：输出猜测
  → 你写下第一个词："于是"...
```

Transformer 在"生成文字"时，做的事**完全一样**。它不是在无中生有，而是在"给定历史信息，预测下一个最可能的词"。

---

## Q、K、V 是什么——大白话解释

每个 token（词）会产生三个向量：

| 向量 | 名字 | 比喻 | 做什么 |
|------|------|------|--------|
| **Q** (Query) | 查询 | 每个人手里拿的问题 | "我想知道什么" |
| **K** (Key) | 索引卡 | 图书馆每本书的目录卡 | "我包含什么信息" |
| **V** (Value) | 书本内容 | 图书馆里放的书本身 | "我的实际内容是什么" |

### Attention 计算：大白话版

Attention 是 Transformer 的核心机制。它做的事用大白话说是：

```
每个 token 拿着自己的 Q（问题），去问所有其他 token 的 K（索引卡）：
"你们谁的内容和我要查的东西最相关？"

然后根据相关程度，从对应的 V（内容）里加权平均取值。
```

举例：处理句子"我爱吃苹果"

```
"吃"这个 token 的 Q 问："谁是主语？"
→ "我"的 K 和它很匹配 → "我"的 V 被赋予高权重
→ "吃"理解了上下文："哦，原来主语是'我'"

"果"的 Q 问："前面说的是什么水果？"
→ "苹"的 K 和它很匹配 → "苹"的 V 被赋予高权重
→ "果"理解了上下文："哦，前面说的是'苹果'"
```

这就是 Transformer 能"理解上下文"的秘密——每个位置都能看到所有其他位置，通过 QKV 的交互找到彼此的关联。

## Transformer 的两阶段：Prefill 和 Decode

当你给 AI 发一段话，AI 处理这段话分两个阶段：

### 阶段 1：Prefill（预处理所有输入）

**干的事**：把整段输入一口气全部处理完。

```
比如你发了 500 个字（token）

500 个 token 要两两算 Attention
= 500 × 500 = 250,000 次计算

每算完一个 token，它的 K 和 V 就被存进缓存
这个"全部并行算一遍、顺便存好"的过程就是 Prefill
```

Prefill 做完两件事：
1. **理解了整段输入的上下文**——每个 token 都知道自己在整段话里的位置和关系
2. **把每个 token 的 K 和 V 存进缓存**——为后面的生成做准备

**费用**：Prefill 的计算成本算在"输入 token 数 × 单价"里。你付的钱覆盖了这个过程。

---

### 阶段 2：Decode（逐个生成输出）

**干的事**：AI 开始"写作文"，一个词一个词往外蹦。

```
生成第 1 个词：
  → 用特殊 token [BOS]（Begin Of Sequence = "开始"）作为输入
  → [BOS] 算自己的 Q，去查询 Prefill 存好的所有 K
  → 结合所有历史 token 的 V，得到一个"上下文向量"
  → 前馈网络算概率分布
  → 抽样输出第 1 个词，比如"于是"

生成第 2 个词：
  → 历史 = 原始输入 + "于是"
  → "于是"算自己的 Q，去查询包括自己在内的所有 K
  → 得到新的上下文向量
  → 前馈网络算概率分布
  → 抽样输出第 2 个词，比如"他"

生成第 3 个词：
  → 同样套路，"他"的 Q 查询整个历史
  → ...
  → 重复，直到输出完整的句子
```

**关键理解**：每次生成新 token，都会回头查整个历史（Prefill 存的所有 K 和 V），然后预测下一个最可能的词。

**费用**：每生成一个 token 算一次 Decode。所有输出的 token 数 × 单价 = 输出费用。

---

## KV Cache：为什么能省算力

### 没有 KV Cache 的代价

每生成一个新 token，如果要重新算整个历史的所有 QKV，那生成 100 个词 = 算 100 遍历史 = 算力爆炸。

### 有 KV Cache 的情况

Prefill 阶段已经算好了所有 K 和 V。Decode 时：

```
新 token 只需要：
  1. 算自己这一个 token 的 Q、K、V（一 handful）
  2. 用自己的 Q 查询历史所有缓存的 K（直接拿，不用重算）
  3. 用得到的权重对历史所有缓存的 V 做加权平均（直接拿，不用重算）

每次只需要算 1 个新 token 的 QKV，而不是 N 个
```

这就是省算力的地方。**缓存的是 K 和 V，不是 Q——Q 每次都要重新算。**

### 在多轮对话里的效果

```
第 1 轮：500 tokens 输入
  → Prefill：500 个 token 全部算一遍 QKV，存进缓存
  → 付 500 tokens 的输入费

第 2 轮：追加了新消息
  → Prefill：新消息的 token 全部算一遍 QKV，存进缓存
  → 历史那 500 个 token 的 QKV 不重算，直接复用缓存
  → 付 500 历史 tokens（缓存价）+ N 新 tokens（全价）

第 3 轮：同样套路
  → 付 历史 tokens（缓存价）+ N 新 tokens（全价）
  → 以此类推
```

**DeepSeek 的定价**：
- Cache Miss（全价）：$0.14 / M tokens
- Cache Hit（缓存价）：$0.028 / M tokens（打 2 折）

## 全流程串起来：一次对话的完整旅程

```
你输入一段文字（比如"帮我写一个加法函数"）

↓

【Prefill 阶段】
把这段文字切成 token
→ 每个 token 算 Q、K、V
→ 两两之间算 Attention（理解上下文）
→ 所有 K 和 V 存进缓存
→ 付 "输入 token 数 × $0.14/M" 的钱

↓

【Decode 阶段】
AI 开始一个字一个字往外蹦：

  生成第 1 个词：
    → [BOS] 拿 Q 查询缓存的 K
    → 得到上下文向量
    → 算概率分布
    → 抽样输出"def"
  生成第 2 个词：
    → "def" 拿 Q 查询缓存 + 新 token 的 K
    → ...
    → 抽样输出"add"
  生成第 3 个词：
    → ...

↓

每生成一个词 = 付一个词的输出费
直到模型输出结束标记，或到达上限

↓

你得到完整回复
```

## 总结：记住这三个核心要点

| 要点 | 内容 |
|------|------|
| **Transformer 在干什么** | 给定历史，预测下一个最可能的词。不是无中生有，是概率预测 |
| **Attention 是什么** | 每个 token 通过 Q（问问题）和 K（给答案）、V（内容）的交互，找到彼此的关联 |
| **Prefill vs Decode** | Prefill = 并行处理所有输入，算 QKV 存缓存；Decode = 逐个生成，每步复用缓存的 K 和 V |

## 附录：和计费文档的关联

如果你读了 `llm-billing-explained.md`，现在可以把这三件事串起来：

```
Prefill 覆盖所有输入 token → 这就是你付的"输入费用"
Decode 逐个生成新 token → 这就是你付的"输出费用"
KV Cache 存的是 K 和 V → 省的是 Prefill 阶段的计算，不直接影响你的账单价格
                              但 DeepSeek 把省出的算力让利给你 → cache hit 定价更低
```
